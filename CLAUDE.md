# Project Overview

**IaC Toolbox Documentation Site** — A Next.js-based technical documentation platform for Infrastructure as Code (Terraform + AWS) tutorials and guides. Built with Fumadocs for content management and rendering.

## Mission

Replace blog-style v0 content with technical reference documentation modeled after k3s/litellm style. Target audience: indie developers, small startup founders, and engineers new to infrastructure-as-code or AWS.

## Data Flow

### Build-Time (Static Generation)
1. **fumadocs-mdx postinstall hook** — Scans `content/docs/`, generates `.source/server.ts` with MDX imports
2. **source.config.ts** — Defines docs collection schema, MDX plugins (Mermaid)
3. **Next.js build** — Compiles MDX to React, generates static pages via `generateStaticParams()`
4. **Output** — Static HTML + hydration bundles

### Runtime (Page Request)
1. Request `/docs/quick-start/index` → `docs/[[...slug]]/page.tsx`
2. `source.getPage(params.slug)` — Fumadocs loader retrieves page metadata + MDX component
3. Render `<DocsPage>` with title, description, TOC, body
4. MDX body rendered with custom components (Mermaid, relative links)
5. Client hydration + PostHog event tracking

### Content Authoring Flow
```
Edit content/docs/*.mdx
  ↓
Run `pnpm postinstall` (or dev server auto-runs)
  ↓
fumadocs-mdx regenerates .source/server.ts
  ↓
Next.js HMR updates page
```

## Key Components

### Documentation Engine

**src/lib/source.ts**
- Exports `source` loader from Fumadocs
- Configures base URL (`/docs`), plugins (Lucide icons)
- Provides `getPageImage()` for OG images
- Provides `getLLMText()` for AI-readable content

**src/app/docs/[[...slug]]/page.tsx**
- Catch-all route for all documentation pages
- Uses `source.getPage(slug)` to fetch content
- Renders with Fumadocs UI components (`DocsPage`, `DocsTitle`, etc.)
- Includes `<Feedback>` component for GitHub issue creation
- Generates metadata for SEO

**content/docs/meta.json**
- Defines sidebar navigation structure
- Order of sections: `quick-start`, `installation`, `cli-reference`, etc.

### MDX Processing

**source.config.ts**
- Configures Fumadocs MDX collections
- Enables `remarkMdxMermaid` for diagram rendering
- Sets frontmatter/meta schemas

**src/mdx-components.tsx**
- Custom component mappings for MDX
- Integrates `createRelativeLink` for intra-doc navigation

**src/components/mermaid.tsx**
- Client-side Mermaid.js rendering
- Supports dark mode theme switching

### Theming & State

**src/contexts/settingsContext.tsx**
- React Context for user preferences
- Persists to localStorage
- Schema: `{ mode: 'light'|'dark'|'system', theme: { styles } }`

**src/components/theme-provider.tsx**
- Wraps app with `next-themes` provider
- Enables system-level dark mode detection

### Analytics

**src/app/posthog-provider.tsx**
- Initializes PostHog client (EU instance)
- Tracks page views, feedback events

**src/components/feedback.tsx**
- Captures user ratings
- Creates GitHub issues via Octokit
- Sends PostHog events

## Design Patterns

### Route Groups
- `(home)` — Landing page isolated from docs layout
- `docs` — Documentation section with sidebar

### Server Components by Default
- All pages are React Server Components
- Client components marked with `'use client'`
- Used for: PostHog, theme provider, settings context

### Catch-All Dynamic Routes
- `[[...slug]]` — Optional catch-all for `/docs/*` paths
- Enables nested documentation hierarchy
- Falls back to 404 via `notFound()` for missing pages

### MDX Component Overrides
- Custom `<a>` for relative links between docs
- Custom Mermaid component for diagrams
- Preserves markdown semantics while adding interactivity

### Type Safety
- TypeScript strict mode
- Fumadocs generates types for frontmatter/meta schemas
- Path aliases prevent relative import hell

## Performance Optimizations

### Static Generation
- All docs pages pre-rendered at build time
- `generateStaticParams()` provides slug list
- No runtime MDX compilation

### Font Loading
- Local fonts (Montserrat) via `next/font/local`
- Preloaded, no FOUT/FOIT

### Image Optimization
- OG images generated per page
- Next.js automatic image optimization for `/public` assets

# Known Constraints

## Fumadocs Limitations
- MDX files must be in `content/docs/` (configured in `source.config.ts`)
- `.source/` is regenerated on every `postinstall` — never edit directly
- Navigation order controlled by `meta.json`, not filesystem order

## Next.js App Router
- All routes under `src/app/` follow filesystem conventions
- Route groups `(name)` don't affect URL structure
- `[[...slug]]` must be in innermost directory for catch-all

## Build Process
- TypeScript must compile before Next.js build
- MDX processing happens before type checking
- Build fails if any MDX file has invalid frontmatter

## External Dependencies
- PostHog requires client-side JS (not SSR-compatible)
- Mermaid diagrams require client-side rendering
- GitHub feedback requires server action + API token

# Troubleshooting

## Build Failures

**"Cannot find module 'fumadocs-mdx:collections/server'"**
- Run `pnpm postinstall` to regenerate `.source/`

**TypeScript errors in MDX files**
- Check frontmatter matches schema in `source.config.ts`
- Ensure `meta.json` files are valid JSON

**ESLint errors**
- Run `pnpm lint:fix` for auto-fixable issues
- Check `.eslintrc` for custom rules

## Runtime Issues

**Dark mode not working**
- Verify `ThemeProvider` wraps app in `layout.tsx`
- Check localStorage for `theme` key

**Mermaid diagrams not rendering**
- Client component boundary required
- Check browser console for Mermaid errors

**404 on docs pages**
- Verify page exists in `content/docs/`
- Check `meta.json` includes page
- Run `pnpm postinstall` to regenerate imports
